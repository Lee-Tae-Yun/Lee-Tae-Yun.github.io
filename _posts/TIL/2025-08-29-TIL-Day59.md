오늘은 Snowy 프로젝트를 진행하며 `HotelGridCell`의 이미지가 미세하게 늦게 뜨고, 스크롤 시 퍼포먼스가 저하되던 문제를 해결한 과정을 기록해본다.

## 🧐 문제 상황
컬렉션 뷰를 빠르게 스크롤할 때, 셀 내부에 있는 호텔 이미지가 즉각적으로 나타나지 않고 아주 짧은 찰나에 하얀 배경이 보였다가 뜨는 현상이 발생했다. 기능적으로는 문제가 없었지만, 사용자 경험(UX) 측면에서는 앱이 '무겁다'는 인상을 줄 수 있는 부분이었다.

## 🔍 원인 분석

### 1. 비싼 마스킹 연산의 반복
이미지 뷰의 상단과 하단 코너 라운드 값이 다른 디자인을 구현하기 위해 `KeyFrameView`에서 `UIBezierPath`를 사용하여 `CAShapeLayer`를 마스킹하고 있었다. 그런데 이 패스를 생성하고 레이어에 적용하는 코드가 `layoutSubviews()`에서 매번 실행되고 있었다. 셀이 스크롤될 때마다 이 복잡한 계산이 메인 스레드에서 반복되면서 병목이 발생했다.

### 2. 불필요한 이미지 재할당
`prepareForReuse()` 시점에 기본 이미지를 매번 할당하고 있었다. 이미지가 이미 올바르게 설정되어 있음에도 불구하고, 셀이 화면에 다시 나타날 때마다 이미지를 새로 로드하고 할당하는 과정에서 미세한 렌더링 지연이 발생했다.

---

## 🛠 해결 방법 (Optimization Steps)

### 1. Path Caching (패스 캐싱)
`layoutSubviews()`가 호출될 때마다 무조건 패스를 새로 그리는 것이 아니라, 뷰의 `bounds`가 변경되었을 때만 패스를 재계산하도록 로직을 수정했다.
- `lastMaskBounds`를 저장해두고 현재 `bounds`와 비교.
- 동일하다면 캐싱된 `cachedMaskPath`를 그대로 사용.

### 2. Layer Rasterization (레이어 래스터화)
복잡한 마스킹 연산은 GPU에 부담을 준다. `shouldRasterize`를 활용해 렌더링 성능을 높였다.
```swift
maskLayer.shouldRasterize = true
maskLayer.rasterizationScale = UIScreen.main.scale
```
이 설정을 통해 복잡한 마스크 레이어 연산 결과가 비트맵으로 캐싱되어, CPU/GPU가 매 프레임마다 패스를 다시 렌더링할 필요가 없게 되었다.

### 3. Smart Image Assignment (스마트 이미지 설정)
셀 재사용 시 이미지가 이미 존재한다면 불필요한 재설정을 건너뛰도록 개선했다.
```swift
if imageBoxView.imageView.image == nil {
  imageBoxView.setImage(image: .storeImg2)
}
```
이 작은 조건문 하나가 스크롤 시 발생하는 이미지 깜빡임(Flicker)을 획기적으로 줄여주었다.

## 📈 결과
최적화 적용 후, 시뮬레이터와 실기기 모두에서 **스크롤이 눈에 띄게 부드러워졌다.** 특히 이미지가 늦게 뜨던 현상이 사라져, 마치 정적인 리스트를 보듯 쾌적한 피드 환경을 구축할 수 있었다.
